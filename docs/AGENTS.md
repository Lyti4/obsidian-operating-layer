# Инструкции для `docs/`

## Область действия

Этот файл применяется ко всему каталогу `docs/`, если более узкий
`AGENTS.md` не уточняет правила. Он наследует все запреты и границы корневого
`AGENTS.md` и не может их ослабить.

## Классы документов

- `canonical` (канонический) — главный текущий источник по теме.
- `active` (активный) — действующее руководство, связанное с каноническим
  источником.
- `historical` (исторический) — прошлое решение или план, не дающий текущих
  полномочий.
- `evidence` (доказательство) — датированный аудит, отчёт или результат
  проверки.

Статус определяется картой `docs/INSTRUCTION_TREE.md`, а не датой изменения
файла. Исторический документ нельзя переписывать так, будто его старый runtime
(реальное состояние сервисов и заданий) является текущим.

## Обязательный след документации

При изменении кода, инструмента, процесса, runtime или инструкции агент обязан:

1. назвать затронутые `canonical` и `active` документы;
2. обновить их в том же срезе; либо
3. записать проверяемую причину `documentation impact: none`.

При добавлении или изменении `tools/*.py` в том же срезе обновляются:

- `docs/tools/INDEX.md`;
- основное семейное руководство под `docs/tools/families/`;
- связанный тест;
- отдельный runbook (пошаговая инструкция с откатом), если операция пишет в
  vault, меняет сервис или утверждённое состояние.

## Источники текущей истины

- полномочия и порядок инструкций — `docs/INSTRUCTION_TREE.md`;
- runtime — `docs/RUNTIME_STATUS.md`;
- безопасность инструментов — `docs/TOOLS_POLICY.md`;
- роли — `docs/agents/`;
- инструменты — `docs/tools/INDEX.md`;
- активная функция Spec Kit — `.specify/feature.json` и соответствующий каталог
  под `specs/`.

## Язык и ссылки

- Пользовательские выводы пишутся по-русски.
- Команды, пути, API, имена полей и стабильные статусы сохраняются буквально.
- При первом употреблении технического термина даётся короткое русское
  объяснение в скобках.
- Каждая обязательная ссылка должна вести к существующему пути репозитория.
- Не копируйте секреты, приватные URL, тексты личных заметок или значения
  учётных данных.

## Проверка

Минимум для документационного среза:

```bash
git diff --check
```

Если меняется проверяемый контракт, код, схема или тест, дополнительно запускайте
узкий тест и `make verify`.
