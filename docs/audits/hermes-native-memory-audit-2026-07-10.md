# Аудит нативной памяти Hermes через Obsidian

Дата проверки: 2026-07-10

Проект: `/home/hermesadmin/work/obsidian-operating-layer`

Режим: repo-only — код, тесты и документация репозитория без изменения живого vault, конфигурации Hermes, служб или расписаний.

## Краткий вердикт

У проекта уже есть сильный safety-core (контур безопасности), проверенные sandbox-проходы семантического поиска и необходимые локальные компоненты. Но Obsidian пока не является родной памятью Hermes:

- Hermes не подключает Obslayer как активный `MemoryProvider` (нативный интерфейс внешней памяти);
- входящие сообщения, файлы, ссылки и голосовые расшифровки автоматически в vault не сохраняются;
- лексический мост читает Markdown, но не выполняет семантический поиск;
- семантические компоненты доказаны отдельными проходами, но постоянный incremental loop (обновление только изменившихся данных) не включён;
- три связанные задачи Hermes cron существуют, но приостановлены;
- локальное распознавание речи установлено, но не является выбранным голосовым маршрутом Hermes.

Итог: это безопасная лаборатория с готовыми строительными блоками, а не завершённая постоянно работающая память. Правильный путь — подключать возможности по одной, сохраняя Obsidian источником истины, а индексы — пересоздаваемыми производными данными.

## Кто

- Дмитрий — владелец данных и принимает решения о живых изменениях.
- Hermes — оркестратор и будущий основной потребитель Obsidian-памяти.
- Obsidian Operating Layer — контролируемая граница между агентом, vault и индексаторами.
- Codex — реализует и проверяет ограниченные изменения репозитория.
- Nanobot — read-only наблюдатель и автор предложений; его связанные задания сейчас приостановлены.
- Внешние MCP/RAG/Graph-компоненты — адаптеры чтения и анализа, но не владельцы записи в vault.

## Зачем

Сделать Obsidian долговременной и понятной человеку памятью Hermes, куда можно безопасно подавать:

- текстовые сообщения и заметки;
- документы и ссылки;
- голосовые сообщения после локальной расшифровки;
- позже — явно подключённые CRM и другие источники.

Hermes должен автоматически находить эти данные обычным, семантическим и графовым поиском, показывать источник и не превращать предположения индекса в подтверждённые факты.

## Почему

Встроенная память Hermes включена, но подходит для короткого профиля, предпочтений и небольших устойчивых фактов. Она не заменяет канонический Markdown-vault:

- пользователь должен видеть и редактировать знания обычными средствами Obsidian;
- у каждой записи должно сохраняться происхождение;
- голосовая расшифровка и исходные вложения требуют отдельного слоя приёма;
- семантический и графовый индексы должны пересоздаваться без потери заметок;
- продвижение материала из входящих записей в канонические заметки требует понятного review-процесса.

## Что пока не работает

- Нет активного Obslayer `MemoryProvider`, поэтому Hermes не делает автоматический prefetch заметок перед ответом.
- Нет автоматического capture-пути из Telegram text/voice/file/link в Obsidian.
- Голосовой маршрут Hermes не переключён на локальный `faster-whisper`; наличие пакета не равно работающей интеграции.
- Расшифровки не сохраняются отдельными Markdown-записями с метаданными источника.
- Лексический мост не использует `bge-m3` и не умеет semantic/hybrid retrieval.
- `codebase-memory-mcp` нельзя считать индексом vault: его текущий corpus — исходники и документы репозитория.
- Нет постоянно работающей очереди incremental indexing, блокировки конкурентного запуска и freshness-отчёта.
- Нет утверждённого live append-only inbox (входящего журнала только с добавлением).
- Нет дедупликации по platform message ID и content hash в живом capture-пути.
- Нет карантина для потенциально секретных или неподдерживаемых вложений.
- Три Obsidian/Nanobot задания не выполняются, пока их отдельно не возобновят.

## Что делаем

### Этап 0 — правда и повторяемость

Первый принятый repo-only срез:

1. Отслеживать лексический мост, тесты и документацию в Git.
2. Не позволять поиску выходить через символическую ссылку за границу выбранного vault.
3. Изолировать test-owned Graphify derived roots, не меняя рабочую incremental-семантику.
4. Исправить runtime-документацию: три задания существуют, но paused.
5. Сохранить этот аудит и карту развития в репозитории.

Доказательство: узкие тесты, повторный Graphify-прогон, `git diff --check` и полный `make verify`.

### Этап 1 — локальный голос

Требует отдельного подтверждения, потому что затрагивает конфигурацию и перезапуск Hermes:

1. Сохранить резервную копию конфигурации без вывода секретов.
2. Переключить STT (speech-to-text; перевод речи в текст) на локальный `faster-whisper`.
3. Задать лимит времени, размера и памяти.
4. Перезапустить только Hermes gateway.
5. Проверить короткое русское голосовое сообщение и журнал ошибок.

Критерий: Hermes получает корректный текст без облачного audio API, а служба остаётся в ресурсных лимитах.

### Этап 2 — нативное read-only вспоминание

Repo-only реализация, затем отдельное подключение конфигурации:

1. Создать provider `obslayer` по нативному Hermes `MemoryProvider` lifecycle.
2. Реализовать `prefetch`, `search`, `get`, `status` без write-tools.
3. Оставить лексический поиск fallback-механизмом.
4. Подключить существующий semantic wrapper только после bounded smoke-теста.
5. Возвращать путь источника и короткий фрагмент, ограничивая объём контекста.

Критерий: в новом Telegram turn Hermes вспоминает заранее подготовленную заметку и показывает путь, не изменяя vault.

### Этап 3 — безопасный входящий конвейер

Сначала только sandbox:

1. Нормализовать text/voice/file/link в capture envelope: время, источник, message ID, session ID, тип, текст/расшифровка, метаданные вложения и hash.
2. Писать записи в изолированную копию inbox.
3. Добавить идемпотентность: повторная доставка не создаёт дубль.
4. Отправлять секретоподобные и неподдерживаемые данные в quarantine/proposal.
5. Проверить текст и голос на disposable sandbox vault.

После доказанной безопасности отдельно утвердить только один live path: `Memory-Vault/Hermes/Inbox/`. Разрешённые действия — create/append; запрещены replace/delete/move.

Критерий: текст и голос создают проверяемые записи, повтор не дублируется, protected paths не меняются.

### Этап 4 — постоянный семантический индекс

1. Использовать принятый read-only путь `obsidian-semantic-mcp` с локальным Ollama `bge-m3`.
2. Индексировать только новые или изменённые inbox/canonical notes.
3. Добавить очередь, lock, timeout, resource preflight и частичный отчёт.
4. Проверить ручной запуск и восстановление после прерывания.
5. Только затем запросить отдельное разрешение на расписание.

Критерий: новая запись находится keyword, semantic и hybrid запросами; изменение одной заметки не перестраивает весь vault; индексатор не меняет fingerprint живых заметок.

### Этап 5 — связи, консолидация и граф

1. Предлагать related notes, backlinks, duplicates и MOC-кандидаты.
2. Объяснять каждую предложенную связь и показывать источники.
3. Оставлять канонизацию proposal-only.
4. Помечать inferred/ambiguous edges как предположения.

Критерий: граф помогает навигации, но не получает права объявлять выводы подтверждёнными фактами.

### Этап 6 — дополнительные источники и интерфейс

Подключать документы, URL, CRM exports, почту и календарь отдельными адаптерами. Для каждого источника обязательны provenance, retry, quarantine, disable и rollback. В Obsidian нужен dashboard для inbox, freshness индекса, failed captures и review queue.

## Границы

- Не выводить и не сохранять секреты, токены, приватные URL и содержимое личных заметок в отчётах.
- Не давать внешнему MCP/RAG/Graph-компоненту прямое право записи.
- Не выполнять автоматические delete, move или rename.
- Не изменять Soul-Vault и канонические заметки без proposal, approval, backup и verify.
- Не возобновлять cron, не менять Hermes config и не перезапускать службу в repo-only срезе.
- Не считать derived index источником истины.
- Не смешивать CRM, VPN, ParserRIba и другие инициативы без явного контракта источника данных.

Целевое разделение:

- Hermes builtin memory — короткие предпочтения и устойчивые факты;
- Obsidian Inbox — новые входящие материалы и расшифровки;
- Canonical Memory-Vault — проверенные решения и состояния проектов;
- Soul-Vault — identity/поведение, только human-gated;
- derived indexes — SQLite, embeddings и graph, которые можно полностью пересоздать.

## Проверка

Для каждого этапа нужны воспроизводимые доказательства:

- тест до исправления должен воспроизводить дефект;
- узкие тесты проходят после минимальной правки;
- повторный запуск не зависит от старых `out/`-артефактов;
- `git diff --check` проходит;
- для code/runtime-изменений проходит `make verify`;
- `git status` содержит только утверждённые repo-only файлы;
- live vault fingerprint не меняется от read-only поиска и индексации;
- runtime/config/service/cron действия выполняются только после отдельного подтверждения и с rollback-планом.

## Следующий шаг

После полного GREEN первого среза следующий полезный шаг — отдельный этап 1: локальный STT с backup конфигурации, ограниченным перезапуском gateway и одним голосовым smoke-тестом. Затем — repo-only реализация read-only `MemoryProvider`. Автоматическую запись в живой inbox включать только после успешного sandbox capture для текста и голоса.

## Термины

- `MemoryProvider` — нативный интерфейс внешней памяти Hermes.
- MCP — протокол подключения инструментов к агенту.
- STT — перевод речи в текст.
- Embedding — числовое представление смысла текста для семантического поиска.
- FTS5 — полнотекстовый поиск SQLite по словам и фразам.
- RAG — поиск подходящих источников перед ответом модели.
- Incremental indexing — обновление индекса только для новых или изменённых файлов.
- Append-only — запись только добавлением, без перезаписи и удаления старого.
- Provenance — сведения о происхождении записи и её источнике.
- Sandbox — изолированная тестовая копия, не затрагивающая живые данные.
- Smoke test — короткая проверка основного рабочего пути.
- Derived index — пересоздаваемый индекс, не являющийся первичным источником данных.
