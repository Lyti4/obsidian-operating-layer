# Спецификация дерева инструкций и документации инструментов

**Статус:** утверждено; дополнено разрешением на `.specify/`

**Дата:** 2026-07-10

**Решение пользователя:** вариант A — иерархия от общего уровня к узкому

**Базовая ветка:** `codex/hermes-native-memory-first-slice`

**Рабочая ветка:** `codex/instruction-tree-docs`

## Краткий вердикт

Проекту нужна одна проверяемая система инструкций вместо набора равноправных
Markdown-файлов. Корневой `AGENTS.md` остаётся главным проектным договором,
узкие `AGENTS.md` уточняют правила только внутри своих каталогов, а
`docs/INSTRUCTION_TREE.md` объясняет человеку и агенту, какой документ за что
отвечает.

Все 58 отслеживаемых Python-файлов из `tools/` должны быть классифицированы в
едином реестре. Для восьми семейств инструментов создаются общие руководства.
Отдельный runbook (пошаговая инструкция для рискованной операции) нужен только
для сложных или опасных сценариев. Исторические отчёты сохраняются как
свидетельства прошлого и не конкурируют с текущими правилами.

## Подтверждённое исходное состояние

- В базовом коммите 115 отслеживаемых Git Markdown-файлов; эта спецификация
  станет 116-м после отдельного разрешения на коммит.
- В `tools/` находятся 58 отслеживаемых Python-файлов.
- Из них 56 являются запускаемыми инструментами, а `_bootstrap.py` и
  `common.py` — внутренними вспомогательными модулями.
- Канонические документы явно называют только часть инструментов; полнота
  покрытия не проверяется автоматически.
- Корневой `AGENTS.md` — единственный полноценный проектный файл инструкций.
- Контракты ролей уже есть в `docs/agents/HERMES.md`, `CODEX.md` и
  `NANOBOT.md`, но над ними нет локального файла области действия.
- `HANDOFF.md`, `ROLE_NOTES.md` и часть документов `docs/spec-kit/` содержат
  устаревшие или исторические формулировки, которые выглядят как текущие.
- Текущий аудит отставания документации проверяет небольшой набор текстовых
  маркеров, но не проверяет наличие инструмента в реестре и его связи с тестом.
- Запуск текущего аудита зафиксировал состояние `lagging`: в `AGENTS.md` нет
  ожидаемых старым валидатором фраз `15 minutes` и
  `bounded read-only/proposal-only`. Это подтверждает расхождение между
  текущими инструкциями и самим проверяющим правилом.
- На момент аудита в репозитории не было `.specify/`. Обновлённая цель
  пользователя от 2026-07-10 явно разрешает создать каталог и шаблоны, поэтому
  дальнейший план использует официальный каркас Spec Kit и сохраняет
  `docs/spec-kit/` как историко-проектный слой.

## Проблема

Сейчас читателю приходится самостоятельно решать, какой документ актуален,
какой исторический и какой относится к конкретному инструменту. При добавлении
нового файла в `tools/` ничто не требует одновременно описать его назначение,
риск, поверхность записи и тест. В результате инструкции могут отставать от
кода, а устаревшая заметка — восприниматься как действующий договор.

## Цели

1. Установить однозначный приоритет инструкций от общего уровня к узкому.
2. Описать все 58 файлов инструментального слоя без необязательного
   дублирования.
3. Разделить текущие правила, рабочие руководства, историю и доказательства.
4. Сделать полноту документации автоматически проверяемой.
5. Сохранить понятный русский слой для владельца проекта и точные английские
   идентификаторы для команд, путей и API.
6. Подготовить основу для будущего read-only MemoryProvider (провайдер памяти
   только для чтения; автоматически подаёт релевантные заметки Hermes), не
   включая его реализацию в эту документационную работу.

## Не входит в эту работу

- установка или включение Hermes MemoryProvider;
- изменение живого Obsidian-хранилища;
- изменение сервисов, cron-задач, авторизации или сетевых настроек;
- массовая переработка кода инструментов;
- переписывание исторических аудитов под сегодняшнее состояние;
- создание отдельного Markdown-файла для каждого из 58 инструментов;
- коммит и отправка ветки без отдельного разрешения владельца.

## Модель полномочий и приоритета

Порядок применения правил:

1. Системные и пользовательские инструкции среды находятся вне репозитория и
   имеют высший приоритет.
2. Ближайший по пути `AGENTS.md` уточняет правила для своего каталога и всех
   вложенных каталогов.
3. Узкий `AGENTS.md` не может ослабить безопасность, границы данных и запреты
   корневого `AGENTS.md`.
4. Контракт конкретной роли уточняет, кто выполняет действие, но не отменяет
   правила каталога.
5. Семейное руководство объясняет общий способ работы группы инструментов.
6. Runbook описывает конкретную операцию и её проверки.
7. Спецификации, аудиты и отчёты служат планом или свидетельством; сами по себе
   они не становятся действующей инструкцией.
8. При конфликте применяется более высокий уровень. Неустранимый конфликт
   останавливает изменение до решения владельца проекта.

## Области действия `AGENTS.md`

### Корневой `AGENTS.md`

Содержит только правила, действующие для всего проекта:

- цель проекта и границы данных;
- порядок полномочий;
- режимы `read-only`, `proposal-only`, `sandbox` и `approved-write`;
- запрет секретов и неутверждённой записи;
- обязательные проверки;
- правила коммита, отправки и изменения живой среды;
- ссылки на карту инструкций, роли и реестр инструментов.

Он не должен превращаться в каталог всех команд.

### `docs/AGENTS.md`

Определяет правила для Markdown-документов:

- обязательный статус документа;
- источник фактов и дата проверки;
- запрет выдавать исторический текст за текущую инструкцию;
- правила ссылок и терминов;
- обязательное обновление реестра при добавлении инструмента;
- структурные блоки для решений и передач контекста.

### `docs/agents/AGENTS.md`

Определяет единый шаблон контрактов ролей:

- назначение роли;
- разрешённые действия;
- запрещённые действия;
- входы и выходы;
- граница записи;
- передача другой роли;
- проверка результата;
- источник текущего runtime-состояния.

### `tools/AGENTS.md`

Определяет правила инструментального слоя:

- безопасные значения по умолчанию;
- явный флаг для разрешённой записи;
- машинно-читаемый результат там, где он нужен автоматизации;
- отсутствие секретов и тел заметок в логах;
- связь с тестом и строкой реестра;
- стандартные коды завершения и понятные ошибки;
- обновление документации в том же изменении, что и инструмент.

### `src/obslayer/AGENTS.md`

Защищает ядро безопасности:

- операции с путями не выходят за разрешённые корни;
- запись допускается только через утверждённый контур;
- dry-run (пробный запуск без записи) остаётся режимом по умолчанию там, где
  это предусмотрено контрактом;
- изменение модели доверия требует отдельного решения и тестов.

### `tests/AGENTS.md`

Фиксирует доказательную дисциплину:

- тесты не зависят от живого vault (хранилища заметок Obsidian);
- временные и производные данные принадлежат конкретному тесту;
- тест записи проверяет также отказ без разрешения;
- новая команда имеет узкий тест контракта;
- нестабильный тест не объявляется успешным без устранения причины.

## Классы документов

Каждый рабочий документ получает один из четырёх статусов:

| Статус | Значение | Может задавать текущие правила |
|---|---|---|
| `canonical` | главный актуальный источник для темы | да |
| `active` | действующее руководство или рабочий план | только в границах ссылки от canonical |
| `historical` | прошлое решение, план или передача контекста | нет |
| `evidence` | датированный аудит, отчёт или результат проверки | нет |

Правило «последний изменённый файл важнее» не применяется. Актуальность
определяется статусом, деревом полномочий и явными ссылками.

## Реестр инструментов

`docs/tools/INDEX.md` становится единственным полным перечнем файлов
`tools/*.py`. Одна строка соответствует одному отслеживаемому Python-файлу,
включая внутренние модули.

Обязательные поля строки:

| Поле | Что фиксирует |
|---|---|
| `tool` | точный путь файла |
| `purpose` | зачем инструмент существует |
| `kind` | `cli` или `internal` |
| `family` | одно основное семейство |
| `mode` | `read-only`, `proposal-only`, `sandbox` или `approved-write` |
| `write_surface` | куда возможна запись; `none` при отсутствии |
| `inputs` | основные входы без секретных значений |
| `outputs` | файлы, JSON, отчёты или код возврата |
| `test` | основной автоматический тест или обоснованное исключение |
| `instruction` | семейное руководство или runbook |
| `spec` | подробная спецификация, если она существует |
| `status` | `active`, `experimental`, `deprecated` или `internal` |

Исключение из теста допускается только для внутреннего модуля, если его
поведение полностью покрывается тестами вызывающих инструментов. Это должно
быть написано в поле `test`, а не оставлено пустым.

## Восемь семейств инструментов

### 1. Hermes memory

Назначение: получение контекста из Obsidian для Hermes.

Начальный участник: `tools/hermes_obslayer_memory.py`. Семейство фиксирует
read-only границу, лимит результатов, таймаут, fail-open (безопасное
продолжение без памяти при сбое) и запрет логирования содержимого заметок.

### 2. Core vault workflow

Назначение: наблюдение, предложение, песочница, утверждённое применение и
проверка.

Примеры: `obsidian_observe.py`, `obsidian_propose.py`, `obsidian_sandbox.py`,
`obsidian_apply.py`, `obsidian_verify.py`, `obsidian_controlled_autonomy.py`.

### 3. Indexing, Graphify and semantic search

Назначение: индексация, embeddings (числовые представления смысла текста),
поиск и проверка производных индексов.

Примеры: `obsidian_indexing_runtime.py`,
`obsidian_graphify_incremental_index.py`,
`obsidian_graphify_embedding_query.py`, `obsidian_semantic_manifest.py`,
`obsidian_rag_graph_adapter.py`.

### 4. Link hygiene

Назначение: обнаружение, оценка и безопасное предложение исправлений ссылок.

Примеры: `obsidian_remaining_link_target_discovery.py`,
`obsidian_remaining_link_triage.py`, `obsidian_candidate_scorer.py`,
`obsidian_standing_safe_link_prefix_policy.py`.

### 5. Operator and control plane

Назначение: очереди, решения оператора, панели обзора, готовность применения и
единые индексы управления.

Примеры: `obsidian_operator_review_queue.py`,
`obsidian_operator_decision_ledger.py`,
`obsidian_unified_control_plane_index.py`,
`obsidian_approved_apply_readiness.py`.

### 6. Hermes, Codex and Nanobot collaboration

Назначение: безопасная передача заданий и доказательств между ролями.

Примеры: `codex_hermes_comm.py`, `hermes_codex_run.py`,
`nanobot_readonly_evidence_gateway.py`, `nanobot_review_packet.py`.

### 7. Reports and evidence

Назначение: отчёты, acceptance bundle (пакет доказательств приёмки),
диагностика и проверка отставания документации.

Примеры: `obsidian_backfill_report.py`,
`obsidian_acceptance_bundle_doctor.py`, `obsidian_diagram_pdf_report.py`,
`obsidian_project_docs_lag_audit.py`.

### 8. Internal support modules

Назначение: общая загрузка окружения и переиспользуемые функции.

Участники: `tools/_bootstrap.py` и `tools/common.py`. Они не должны
представляться пользователю как CLI, но обязаны присутствовать в реестре и
указывать, какими тестами вызывающих инструментов покрываются.

Каждый инструмент относится к одному основному семейству. Дополнительные связи
оформляются ссылками, чтобы не появлялись две конкурирующие инструкции.

## Правила семейных руководств

В `docs/tools/families/` создаётся восемь файлов. Каждый содержит:

1. цель семейства;
2. владельца или основную роль;
3. модель данных и границы;
4. режимы безопасности;
5. общий поток выполнения;
6. таблицу входящих инструментов;
7. общие проверки;
8. ссылки на сложные runbook;
9. известные ограничения;
10. понятия с коротким русским объяснением.

Семейное руководство не копирует полную справку `--help`. Точные параметры
остаются в коде и проверяются тестом CLI-контракта.

## Когда нужен отдельный runbook

Отдельный runbook создаётся, если операция:

- пишет в vault или меняет утверждённое состояние;
- затрагивает сервис, расписание, авторизацию или внешний провайдер;
- имеет несколько необратимых либо зависимых шагов;
- требует резервной копии или отката;
- передаёт управление между Hermes, Codex и Nanobot;
- имеет известный опасный отказ, который нельзя объяснить одной строкой в
  семейном руководстве.

Обязательная структура: предусловия, права, dry-run, команда, ожидаемое
доказательство, остановка при ошибке, откат, секреты и послепроверка.

## Контракты ролей

- `HERMES.md` описывает оркестрацию, запрос контекста и принятие решений.
- `CODEX.md` описывает изменение репозитория, тестирование и подготовку
  предложений.
- `NANOBOT.md` описывает ограниченные read-only и reviewer-задачи.
- Текущее состояние сервисов и расписаний хранится в
  `docs/RUNTIME_STATUS.md`, а не копируется в каждый контракт.
- Любое упоминание команды сверяется с реальным `--help` или тестом.
- Контракт роли ссылается на семейное руководство, но не дублирует его.

## Язык и термины

- Пользовательские объяснения и краткие выводы пишутся по-русски.
- Имена файлов, команд, флагов, API и статусов сохраняются на английском.
- При первом употреблении технического термина даётся короткое русское
  объяснение в скобках.
- Один и тот же термин не переводится заново в каждом абзаце.
- Команды приводятся буквально; их перевод помещается рядом, а не внутрь
  команды.

## Работа с устаревшими документами

- `HANDOFF.md` становится короткой текущей точкой входа со ссылками на
  `PROJECT_OVERVIEW.md`, `RUNTIME_STATUS.md`, активную спецификацию и следующий
  утверждённый шаг.
- `ROLE_NOTES.md` получает статус `historical` и явное перенаправление на
  `docs/agents/`.
- Старые документы `docs/spec-kit/` сохраняют номер, дату и историческое
  назначение. Устаревшие runtime-утверждения получают заметную ссылку на
  `docs/RUNTIME_STATUS.md`.
- Аудиты не переписываются. Новый факт оформляется новым датированным аудитом.
- `docs/operator-guide.md` различает наличие определения задания, его текущее
  состояние и факт установки проектом.

## Автоматическая проверка полноты

`obsidian_project_docs_lag_audit.py` расширяется от проверки отдельных фраз до
проверки структуры.

Проверка должна завершаться ошибкой, если:

1. отслеживаемый `tools/*.py` отсутствует в `docs/tools/INDEX.md`;
2. в реестре указан несуществующий файл;
3. отсутствует обязательное поле строки;
4. CLI не имеет теста либо явного проверяемого исключения;
5. ссылка на инструкцию, тест, спецификацию или runbook не существует;
6. значение `kind`, `family`, `mode` или `status` не входит в разрешённый
   словарь;
7. новый инструмент добавлен без обновления реестра;
8. внутренние `_bootstrap.py` или `common.py` ошибочно представлены как CLI;
9. canonical-документы ссылаются на отсутствующий файл;
10. активный документ выдаёт исторический runtime-факт за текущий без ссылки
    на `docs/RUNTIME_STATUS.md`.

Старые текстовые маркеры сначала сопоставляются с действующими правилами.
Актуальные сохраняются, устаревшие заменяются структурными проверками. Цель —
проверять договор, а не случайную английскую фразу.

## План миграции

### Фаза 0. Зафиксировать решение

Установить официальный локальный каркас `.specify/` версии 0.12.0 с Bash-
сценариями для Linux-сервера, принять проектную конституцию и создать feature-
каталог `specs/001-instruction-tree-tool-documentation/`. Затем перенести
утверждённые требования в `spec.md` и сгенерировать связанные артефакты.

### Фаза 1. Создать каркас полномочий

Обновить корневой `AGENTS.md`; добавить `docs/AGENTS.md`,
`docs/agents/AGENTS.md`, `tools/AGENTS.md`, `src/obslayer/AGENTS.md`,
`tests/AGENTS.md` и `docs/INSTRUCTION_TREE.md`.

### Фаза 2. Покрыть инструментальный слой

Создать полный `docs/tools/INDEX.md` и восемь семейных руководств. Все 58
файлов классифицируются в одном изменении, чтобы не было промежуточного ложного
состояния «полное покрытие».

### Фаза 3. Обновить рабочие Markdown-инструкции

Привести контракты ролей, `HANDOFF.md`, `ROLE_NOTES.md`, операторское
руководство, runtime-ссылки и активные спецификации к новой иерархии.
Исторические доказательства сохраняются.

### Фаза 4. Усилить валидатор

Добавить структурный разбор реестра, проверку ссылок и автоматические тесты.
Новый тест сначала доказывает отсутствие инструмента в реестре, затем проходит
на полном наборе.

### Фаза 5. Провести полную проверку

Запустить узкие тесты документационного валидатора, проверку ссылок,
`git diff --check` и полный `make verify`. Отдельно проверить, что изменения не
касаются vault, сервисов, cron, авторизации и секретов.

## Критерии приёмки

- Из любого каталога однозначно определяется применимый `AGENTS.md`.
- `docs/INSTRUCTION_TREE.md` объясняет приоритет без знания кода.
- Все 58 файлов `tools/*.py` представлены ровно по одному разу в реестре.
- У каждого CLI есть семейство, режим риска, поверхность записи, тест и
  инструкция.
- Два внутренних модуля явно помечены как `internal`.
- Восемь семейных руководств покрывают весь реестр.
- Рискованные операции имеют runbook с откатом и доказательством.
- Контракты Hermes, Codex и Nanobot не противоречат корневым границам.
- `HANDOFF.md` и `ROLE_NOTES.md` больше не выглядят самостоятельными текущими
  источниками истины.
- Исторические документы заметно отделены от активных.
- Добавление незадокументированного инструмента ломает автоматическую проверку.
- Все ссылки существуют, `git diff --check` и `make verify` проходят.
- В изменениях нет секретов, содержимого личных заметок и живых runtime-правок.

## Границы безопасности

- Работа выполняется только в репозитории.
- Чтение живого vault для документационной миграции не требуется.
- Установка провайдера, изменение Hermes config, перезапуск сервисов и
  включение cron требуют отдельного согласования.
- Коммит, push (отправка коммита в GitHub) и pull request (запрос на слияние)
  требуют отдельного разрешения владельца.
- Секретные URL, токены, ключи, имена пользователей и тексты заметок не
  попадают в документацию или тестовые данные.

## Откат

Документационная миграция выполняется в отдельной ветке. До слияния откат — это
отказ от ветки. После слияния откат выполняется обычным обратным коммитом по
явному списку файлов. Живое состояние Hermes и Obsidian не меняется, поэтому
операционный откат не нужен.

## Рассмотренные альтернативы

### Один большой `AGENTS.md`

Отклонено: он смешивает проектные границы, команды инструментов и правила
тестов, быстро устаревает и плохо показывает локальную область действия.

### Отдельный документ для каждого инструмента

Отклонено: 58 файлов создадут повторения и увеличат стоимость синхронизации.
Реестр и семейные руководства дают полное покрытие меньшим числом источников.

### Оставить только существующие документы и улучшить поиск

Отклонено: поиск находит текст, но не устанавливает полномочия, актуальность и
обязательное покрытие нового инструмента.

### Установить стандартный `.specify/`

Принято обновлённой целью пользователя 2026-07-10. Используются официальные
шаблоны версии 0.12.0 и Bash-сценарии для Linux. Старые `docs/spec-kit/` не
удаляются: они остаются историческими решениями и источниками доказательств,
а новые активные функции ведутся под `specs/`.

## Структурные кирпичики для других агентов

### Кто

Владелец проекта утверждает границы и внешние действия. Hermes оркестрирует,
Codex меняет и проверяет репозиторий, Nanobot выполняет ограниченные
read-only/reviewer-задачи. Узкие полномочия всегда наследуют корневые запреты.

### Зачем

Чтобы любой агент находил актуальное правило, понимал риск инструмента и не
принимал исторический отчёт за разрешение на действие.

### Почему

Текущий набор документов велик, но не образует проверяемую иерархию, а 58
инструментальных файлов не имеют единого полного реестра.

### Что делаем

Строим вложенное дерево `AGENTS.md`, карту приоритета, полный реестр, восемь
семейных руководств, точечные runbook и структурную автоматическую проверку.

### Границы

Только репозиторий и документация. Без живого vault, сервисов, cron,
авторизации, провайдеров, секретов и неутверждённой публикации.

### Проверка

Полнота 58 из 58, существование ссылок, тест валидатора, `git diff --check`,
`make verify` и ручная проверка конфликтов полномочий.

### Следующий шаг

Создать каркас `.specify/` и feature-каталог, затем подготовить отдельный план
реализации, чек-лист качества и зависимые задачи. Миграцию выполнять небольшими
проверяемыми срезами.

### Термины

- `AGENTS.md` — файл правил для агента в каталоге и ниже.
- CLI — интерфейс командной строки; запускаемый инструмент.
- Registry — реестр; полный проверяемый список инструментов.
- Runbook — пошаговая инструкция безопасной операции.
- Read-only — режим только чтения без изменения источника.
- Proposal-only — режим создания предложения без применения.
- Sandbox — изолированная область для безопасной проверки.
- Approved-write — запись только после явного утверждения.
- Runtime — реально работающее состояние сервисов и заданий.
- Evidence — доказательство: тест, отчёт, журнал или проверяемый вывод.
